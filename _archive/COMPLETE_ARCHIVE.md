## Complete Archive

開発中のシステム（v2.5時点）が抱える欠陥・課題、解決アイデア、そして現実的批判を網羅的に整理。

# 📍 1. 現状の改善点・不足点リスト

## 🏗 アーキテクチャの構造的欠陥 (Fragility)

| 課題 | 詳細 |
|------|------|
| **依存の連鎖** | MacroDroid → Termux → Python → ngrok → Cloud。1箇所でもコケると全停止 |
| **コールドスタート** | Pre-Fetch謳いながら、サーバーがスリープなら即応性ゼロ |
| **単一障害点** | 個人のデバイス設定依存。機種変・OSアップデートで崩壊 |

## 🛡 セキュリティとプライバシーの穴

| 課題 | 詳細 |
|------|------|
| **正規表現の限界** | `090-xxxx` は隠せても「A社のBプロジェクト」は検知不能 |
| **オンデバイス未活用** | 通信経路上のリスク、処理遅延が残る |
| **復元リスク** | タグ置換後、AIが構造を変えると復元不可能→Data Corruption |

## 🧠 「空気読み」の解像度不足

| 課題 | 詳細 |
|------|------|
| **コンテキストの粗さ** | パッケージ名だけで判断。「役員報告」vs「雑談」を区別不能 |
| **画面情報の欠落** | 「誰と、どんな話題で」という真の文脈が見えていない |

## 👤 パーソナライズの浅さ

| 課題 | 詳細 |
|------|------|
| **書き癖の再現性欠如** | 「いつもの感じで」が通じない |
| **学習ループの欠如** | ユーザーの修正を学ばず、同じミスを繰り返す |

## 👆 UXの摩擦

| 課題 | 詳細 |
|------|------|
| **操作の手間** | 選択→メニュー→タップの3アクション必要 |
| **Undo Friction** | 生成は一瞬だが、誤爆時のリカバリーが困難 |
| **視覚的ノイズ** | オーバーレイがコンテンツを隠して邪魔になる |

## ⏳ 時間と育成の断絶

| 課題 | 詳細 |
|------|------|
| **インキュベーター不在** | アイデアを寝かせて育てる機能がない |
| **能動性の欠如** | AI側からの提案がない（常にコピー待ち） |

# 💡 2. 拡散的な視点 (Divergent Ideas)

## UX・インターフェースの革新

| アイデア | 概要 |
|---------|------|
| **カスタムキーボード化** | IMEラッパーとして実装。変換キー感覚でAI介入 |
| **Invisible Interface** | トリガーワード（`;;fix`）で魔法のように書き換え |
| **Pre-Emptive Action** | 画面監視で返信必要時に通知領域へ返信案提示 |
| **Haptic Language** | バイブパターンで処理完了/エラーを通知 |

## 機能・処理の拡張

| アイデア | 概要 |
|---------|------|
| **マルチモーダル** | 画像→テキスト化、音声→要約 |
| **Action Intent** | テキストでなく次の行動（カレンダー登録等）を返す |
| **インキュベーター** | アイデアを放り込むと毎日育てて週末レポート |
| **文体ガチャ** | たまにシェイクスピア風、大阪弁で返信 |

## アーキテクチャ・構造の進化

| アイデア | 概要 |
|---------|------|
| **Local Function Call** | WASMやローカルソケットでレイテンシゼロ |
| **Spatial Context** | Wi-Fi SSID/Bluetoothから場所判定→自動設定 |
| **Context Ballast** | 履歴やアプリ情報を「重り」としてプロンプトに付与 |

## 価値・意味の転換

| アイデア | 概要 |
|---------|------|
| **Cognitive Offloading** | 「時短」でなく「覚えなくていい安心感」を提供 |
| **語学学習ツール** | 修正案+なぜ直したかの解説（Mini-Coach） |
| **コスト予報** | 実行前に「約◯円かかります」と見積もり |

# 🛑 3. 収束的な視点 (Convergent Criticism)

## セキュリティとプライバシーの壁

| 批判 | 詳細 |
|------|------|
| **Detokenization Fallacy** | タグ置換は復元不可能→データ消失・誤復元のリスク |
| **Spyware Architecture** | 常時監視はマルウェア扱い、OS規制対象 |

## 運用と品質の現実

| 批判 | 詳細 |
|------|------|
| **パッチワークの脆さ** | OSアップデート一発で動かなくなる |
| **ROI Delusion** | 個人データでの学習は手間の割に効果薄い |

## UXの阻害要因

| 批判 | 詳細 |
|------|------|
| **触覚の過信** | 振動は「情報」でなく「不安」を与えるノイズ |
| **視覚的ノイズ** | 常時オーバーレイは邪魔→最終的にオフにされる |
| **通信のゆらぎ** | オフライン耐性がないと信頼されない |

---

# ✅ 4. 設計の修正案（現実解）

| 項目 | 現実解 |
|------|--------|
| **タグ置換** | 廃止→「個人情報検知＆警告」に留める |
| **常時監視** | 撤回→「明示的な共有操作」に限定 |
| **育てる機能** | 「選ばせる」方式（3案から選択）が満足度高い |

# 🎯 6. v3.0 開発方針

**「魔法のようなUX」から「毎日使っても壊れない道具」へ**

1. **安全第一:** 自動復元廃止、警告ダイアログ
2. **安定第一:** 常時監視廃止、明示的トリガー
3. **シンプル第一:** 育てるより選ばせる
